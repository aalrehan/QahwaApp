WHY `expo install` vs `npm install`:
  `expo install` checks each package against your installed Expo SDK and picks
  the version Expo has tested as compatible. For `expo-*` packages and known
  React Native libs, this avoids "your version of X doesn't work with SDK 54"

  through to npm, so we use npm directly there.

STEP 5 — SUPABASE CLIENT
================================================================================

FILE: C:\Users\aalre\Projects\qahwa\lib\supabase.ts

CONTENTS:
  import 'react-native-url-polyfill/auto';

  import AsyncStorage from '@react-native-async-storage/async-storage';
  import { createClient } from '@supabase/supabase-js';

  const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL!;
  const supabaseAnonKey = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!;

  export const supabase = createClient(supabaseUrl, supabaseAnonKey, {
    auth: {
      storage: AsyncStorage,
      autoRefreshToken: true,
      persistSession: true,
      detectSessionInUrl: false,
    },
  });

WHY EACH LINE:
  1. `react-native-url-polyfill/auto` — side-effect import that patches the
     URL global. Must come before the Supabase client is used or it crashes
     parsing the Supabase URL.
  2. `process.env.EXPO_PUBLIC_*` — Expo only exposes env vars to client code
     if prefixed with EXPO_PUBLIC_. The non-null assertion `!` is intentional:
     if these are missing, we want a clear crash at boot, not a confusing
     query failure later.
  3. `storage: AsyncStorage` — by default Supabase uses localStorage which
     doesn't exist on native. AsyncStorage is the RN equivalent.
  4. `detectSessionInUrl: false` — only relevant for OAuth web redirects.
     On native, deep links are handled differently. Leaving this on causes
     warnings.

DECISIONS MADE:
  - Singleton export: `export const supabase = ...` runs once at module load
    and is reused across all imports.
  - AsyncStorage (plaintext) chosen over expo-secure-store (encrypted) for
    now. Matches Supabase's default recommendation. expo-secure-store IS
    installed (Step 2) but not yet wired up. Future task: write a custom
    storage adapter that proxies to SecureStore for stronger token security.

SECURITY NOTE:
  The anon key is meant to be public — it's Row-Level-Security at the database
  level that protects your data, not the key itself. Never put a service-role
  key into an EXPO_PUBLIC_* var.

FILE: C:\Users\aalre\Projects\qahwa\app\index.tsx (NEW)
----------------------------------------
Contents:
  import { useEffect, useState } from 'react';
  import { Text, View } from 'react-native';

  import { supabase } from '@/lib/supabase';
  import { theme } from '@/lib/theme';

  type ConnectionStatus =
    | { state: 'connecting' }
    | { state: 'connected'; count: number }
    | { state: 'failed'; message: string };

  export default function HomeScreen() {
    const [status, setStatus] = useState<ConnectionStatus>({ state: 'connecting' });

    useEffect(() => {
      let cancelled = false;

      (async () => {
        try {
          const { count, error } = await supabase
            .from('badges')
            .select('*', { count: 'exact', head: true });

          if (cancelled) return;

          if (error) {
            setStatus({ state: 'failed', message: error.message });
            return;
          }

          setStatus({ state: 'connected', count: count ?? 0 });
        } catch (err) {
          if (cancelled) return;
          const message = err instanceof Error ? err.message : String(err);
          setStatus({ state: 'failed', message });
        }
      })();

      return () => {
        cancelled = true;
      };
    }, []);

    return (
      <View
        className="flex-1 items-center justify-center px-8"
        style={{ backgroundColor: theme.colors.bg }}
      >
        <Text
          style={{
            color: theme.colors.brown,
            fontSize: 96,
            fontWeight: '700',
            textAlign: 'center',
          }}
        >
          قهوة
        </Text>

        <Text
          style={{
            color: theme.colors.brown,
            fontSize: 18,
            fontStyle: 'italic',
            letterSpacing: 4,
            marginTop: 12,
          }}
        >
          QAHWA
        </Text>

        <View className="mt-16">
          <StatusIndicator status={status} />
        </View>
      </View>
    );
  }

  function StatusIndicator({ status }: { status: ConnectionStatus }) {
    if (status.state === 'connecting') {
      return (
        <Text style={{ color: theme.colors.dim, fontSize: 13 }}>
          Connecting…
        </Text>
      );
    }

    if (status.state === 'connected') {
      return (
        <Text style={{ color: '#2D7A3E', fontSize: 13 }}>
          ● Supabase connected ({status.count} badges)
        </Text>
      );
    }

    return (
      <Text style={{ color: '#B33A3A', fontSize: 13, textAlign: 'center' }}>
        ● Connection failed: {status.message}
      </Text>
    );
  }

KEY DESIGN CHOICES IN THIS FILE:

(a) Discriminated union for ConnectionStatus
    Using a union type means each state carries only the data it needs
    (count for connected, message for failed). TypeScript narrows correctly
    in the StatusIndicator switch.

(b) `cancelled` flag in useEffect
    Standard React pattern. If the user navigates away before the Supabase
    query resolves, we don't try to setState on an unmounted component
    (which would throw a warning in dev and waste a render).

(c) `count: 'exact', head: true`
    Asks Postgres for the row count without sending the actual rows over
    the wire. Way cheaper than `.select('*')` then `.length`. Perfect for
    a smoke test where we just need a number.

(d) try/catch wraps the await
    The `error` from Supabase only catches database-level errors (bad SQL,
    RLS denials, table not found). It does NOT catch network failures
    ("TypeError: Network request failed" when offline). The try/catch
    catches those.

(e) Hybrid styling
    NativeWind classes for layout: flex-1, items-center, justify-center,
    px-8, mt-16
    Inline `style` for theme colors and typography: backgroundColor, color,
    fontSize, fontWeight
    This was deliberate — NativeWind classes can't reference theme.colors.*
    at runtime (they'd need to be in tailwind.config.js). Keeping moving
    parts to a minimum until the foundation was verified.

(f) Status colors hard-coded inline
    Green (#2D7A3E) and red (#B33A3A) are not in our theme tokens. Future
    enhancement: add `success` and `error` to lib/theme.ts.

THE CHOICE WE FACED:
  Two reasonable paths from this state:

  Option A — Two commits, no history rewrite
    f37b709  Initial commit (scaffold)
    NEW      Foundation: Expo + TypeScript + ... (our work)
    Pros: Idiomatic git, additive, preserves the scaffold commit as a
          baseline.
    Cons: The first commit was written by Expo CLI, not by us.

  Option B — Single first commit (squash)
    NEW      Foundation: ... (everything, scaffold + our work in one commit)
    Pros: Matches user's literal request ("first commit covering everything
          we built today"). Clean origin point. No "Initial commit" they
          didn't write.
    Cons: Slightly destructive — orphans the scaffold's commit (eventually
          garbage-collected by git). Safe because nothing was pushed.

  USER CHOSE: Option B.

PRE-COMMIT SAFETY VERIFICATION (before any destructive action):
  - Read .gitignore — confirmed `.env` was on line 34 (added by us in Step 6)
  - Ran `git check-ignore -v .env` → returned `.gitignore:34:.env  .env`
    (proof that the .env file would be ignored, not committed)
  - Ran `git ls-files | grep -E "\.env|secret|key"` → empty result
    (proof that no secret-named files were already tracked)

EXECUTION SEQUENCE:

  (a) Orphan the scaffold's initial commit
      $ git update-ref -d HEAD
      Result: HEAD un-set. `git status` now shows "No commits yet".
      The scaffold commit f37b709 still exists in git's object database
      but no branch points to it — orphaned. Will be GC'd eventually.

  (b) Reset the index to match current working tree
      First attempt:
        $ git rm -r --cached . > /dev/null && git add .
        ERROR: "files have staged content different from both the file and
        the HEAD" — git was protective of files we'd modified locally.
      Fixed with -f flag:
        $ git rm -rf --cached . > /dev/null && git add .
        SUCCESS — index now contains exactly what's in the working tree.
        Note: -f here is safe; it only forces removal from the INDEX, never
        touches the working tree.

      CRLF WARNINGS:
      Git printed ~21 warnings like:
        "warning: in the working copy of 'X', LF will be replaced by CRLF
         the next time Git touches it"
      These are informational only — git is normalizing line endings on
      Windows (it stores LF in the repo, checks out CRLF). Not a problem.

      Final index state: 31 files staged, including:
        ✓ .env.example (placeholders only)
        ✓ .gitignore (with .env on line 34)
        ✓ All app/, lib/, config files we wrote
        ✓ scaffold's assets/ images
        ✓ scaffold's .vscode/ extension recommendations

        ✗ .env  (gitignored)
        ✗ node_modules/  (gitignored)
        ✗ .expo/  (gitignored)
        ✗ expo-env.d.ts  (gitignored — auto-generated by Expo)
        ✗ The deleted (tabs)/, components/, hooks/, constants/  (deleted
          from working tree before staging)

  (c) Rename the local branch from `master` to `main`
      $ git branch -m master main
      WHY: Expo's scaffold creates the branch as `master`, but GitHub's
      default for new repos is `main`. If you push as `master`, GitHub
      accepts it but you end up with a `master` branch and an empty default

  5. DO NOT check any of these:
     - Add a README file        ✗

     - Choose a license         ✗

  6. Click "Create repository"

WHAT THE USER ACTUALLY DID:

  capital A, "App" suffix — DIFFERENT from the local folder name `qahwa`).

FIRST PUSH ATTEMPT — FAILED:
  Commands run by the user (using the URL we initially guessed):
    $ git remote add origin https://github.com/aalrehan/qahwa.git
    $ git push -u origin main

  Output:
    info: please complete authentication in your browser...
    remote: Repository not found.
    fatal: repository 'https://github.com/aalrehan/qahwa.git/' not found

  DIAGNOSIS:
  The "info: please complete authentication..." line means Git Credential
  Manager opened a browser auth flow and authenticated successfully —
  GitHub knew who the user was. The "Repository not found" then meant:
  EITHER the repo doesn't exist OR the authenticated account can't see it.
  Common causes (in rough order of likelihood):
    1. Repo wasn't created yet (forgot the click)
    2. Repo name doesn't match what we set in `git remote add`
    3. Repo was created under an organization, not personal account
    4. Wrong account signed into the browser auth flow

  HOW WE DEBUGGED:
  Asked user to verify in browser:
    - Go to github.com → click profile → confirm actual username
    - Click "Your repositories" → confirm the repo exists and what it's
      named
  User pasted a screenshot showing the repo at github.com/aalrehan/QahwaApp.
  Diagnosis: cause #2 — name mismatch (`QahwaApp` vs assumed `qahwa`).

THE FIX (one command + retry):
  $ git remote set-url origin https://github.com/aalrehan/QahwaApp.git
  $ git push -u origin main

  WHY `git remote set-url` and not `git remote remove origin` + add:
  set-url is the right tool when you just need to change the URL of an
  existing remote. Less typing, no chance of forgetting to re-add.

SUCCESSFUL PUSH OUTPUT:
  info: please complete authentication in your browser...
  Enumerating objects: 39, done.
  Counting objects: 100% (39/39), done.
  Delta compression using up to 8 threads
  Compressing objects: 100% (36/36), done.
  Writing objects: 100% (39/39), 676.05 KiB | 3.65 MiB/s, done.
  Total 39 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
  To https://github.com/aalrehan/QahwaApp.git
   * [new branch]      main -> main
  branch 'main' set up to track 'origin/main'.

WHAT EACH PIECE OF THAT OUTPUT MEANS:
  - "Enumerating objects: 39" — git counted everything to send: the commit
    object, the tree object, 31 file blobs, plus subdir tree objects = 39.
  - "Compressing objects: 100% (36/36)" — the 36 unique blobs/trees got
    compressed (the commit + a few others were tiny enough to skip).
  - "676.05 KiB | 3.65 MiB/s" — that's the total wire transfer size and
    speed. Mostly the asset images and package-lock.json.
  - "Total 39 (delta 0)" — first push, so no delta compression possible
    (you only get delta savings on subsequent pushes that share history).
  - "* [new branch] main -> main" — created the `main` branch on the remote.
  - "branch 'main' set up to track 'origin/main'" — the -u flag set the
    upstream. Future plain `git push` and `git pull` know what to do.

POST-PUSH STATE:
  - Local branch: main, commit 199aca2, tracks origin/main
  - Remote: origin → https://github.com/aalrehan/QahwaApp.git
  - Repo on GitHub: 31 files, 1 commit, Private, default branch `main`

COSMETIC NAMING MISMATCH (worth noting):
  Local folder:      qahwa/
  Remote repo:       QahwaApp
  This is fine — git doesn't care about names matching. Two cosmetic
  options if it bothers anyone:
    - Rename the GitHub repo: Settings → Repository name → "qahwa" → Save.
      GitHub keeps redirects from the old URL. Then update the local
      remote: git remote set-url origin https://github.com/aalrehan/qahwa.git
    - Rename the local folder to match. More disruptive.
    - Or leave it. ("App" suffix is a common convention.)

LESSONS WORTH KEEPING:
  - The "Repository not found" error after successful auth nearly always
    means the repo URL is wrong. Don't assume the user's GitHub handle;
    verify it.
  - GitHub repo URLs are case-sensitive in some contexts and not in others.
    Don't rely on case-insensitive matching — use the exact URL.
  - Never `--force` push as a workaround for confusing errors. Diagnose
    first.
