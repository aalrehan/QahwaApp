CONTEXT
--------------------------------------------------------------------------------
Qahwa is an Arabic-first social coffee diary mobile app for Saudi specialty
coffee enthusiasts. The foundation was built in prior sessions: Expo SDK 54,
React Native 0.81, TypeScript, Expo Router 6, NativeWind v4, Supabase JS
client, and a single home screen that renders the قهوة brand mark with a

GOAL
--------------------------------------------------------------------------------
Move from "placeholder bold sans" to actual Naskh calligraphy on the brand
mark. Establish a proper typography system supporting four roles:
  - Arabic display      (headings)
  - Arabic body         (paragraphs, UI text)
  - Arabic decorative   (logo only — Naskh script)
  - English display     (italic serif for ornament)

Arm the app for right-to-left layout so all future UI assumes RTL by default.

LESSONS WORTH KEEPING:
  - Filename casing matters absolutely. expo-font matches files by exact
    string key. One-character typos in case (e.g., `IBMplexSansArabic-Bold`)
    fail silently — font falls back to system, no error thrown.
  - Tajawal is tiny per file (~56 KB) because it's single-script optimized.
  - Amiri files are large (~400 KB) because Naskh ships extensive
    contextual ligatures (the script connects letters in dozens of ways
    depending on position; each glyph variant is stored).
  - 2.3 MB for 10 fonts is reasonable for typography this central to brand
    identity. For comparison: a single React Native app's JS bundle is
    typically 5–15 MB.

  import '../global.css';

  import { useFonts } from 'expo-font';
  import { Stack } from 'expo-router';
  import * as SplashScreen from 'expo-splash-screen';
  import { StatusBar } from 'expo-status-bar';
  import { useEffect } from 'react';
  import { I18nManager } from 'react-native';
  import 'react-native-reanimated';

  SplashScreen.preventAutoHideAsync();

  I18nManager.allowRTL(true);
  if (!I18nManager.isRTL) {
    I18nManager.forceRTL(true);
  }

  export default function RootLayout() {
    const [fontsLoaded, fontError] = useFonts({
      'IBMPlexSansArabic-Regular':  require('../assets/fonts/IBMPlexSansArabic-Regular.ttf'),
      'IBMPlexSansArabic-Medium':   require('../assets/fonts/IBMPlexSansArabic-Medium.ttf'),
      'IBMPlexSansArabic-SemiBold': require('../assets/fonts/IBMPlexSansArabic-SemiBold.ttf'),
      'IBMPlexSansArabic-Bold':     require('../assets/fonts/IBMPlexSansArabic-Bold.ttf'),
      'Tajawal-Regular':            require('../assets/fonts/Tajawal-Regular.ttf'),
      'Tajawal-Medium':             require('../assets/fonts/Tajawal-Medium.ttf'),
      'Tajawal-Bold':               require('../assets/fonts/Tajawal-Bold.ttf'),
      'Amiri-Regular':              require('../assets/fonts/Amiri-Regular.ttf'),
      'Amiri-Bold':                 require('../assets/fonts/Amiri-Bold.ttf'),
      'CormorantGaramond-Italic':   require('../assets/fonts/CormorantGaramond-Italic.ttf'),
    });

    useEffect(() => {
      if (fontsLoaded || fontError) {
        SplashScreen.hideAsync();
      }
    }, [fontsLoaded, fontError]);

    if (!fontsLoaded && !fontError) {
      return null;
    }

    return (
      <>
        <Stack screenOptions={{ headerShown: false }} />
        <StatusBar style="dark" />
      </>
    );
  }

WHY EACH PIECE:

(1) `SplashScreen.preventAutoHideAsync()` AT TOP-LEVEL (not in a hook)
    Without this, the native splash auto-hides as soon as React first
    renders — which happens BEFORE fonts finish loading. Result: user sees
    splash → flash of unstyled (system font) screen → flicker as fonts
    swap in. Calling preventAutoHide BEFORE React mounts (via top-level
    side effect) keeps the splash up until we explicitly hide it.

(2) `useFonts({ ... })` HOOK
    Returns [fontsLoaded, fontError]. fontsLoaded becomes true when ALL
    fonts in the dictionary have loaded; fontError becomes truthy if any
    fail. The string keys (e.g., 'IBMPlexSansArabic-Bold') are the names
    used as `fontFamily` values in components. Convention: match the
    filename (without .ttf extension).

(3) `useEffect` THAT HIDES SPLASH
    Hides on either success or error. If a font fails, we'd rather show
    the app with system fallback than a frozen splash forever.

(4) `if (!fontsLoaded && !fontError) return null;`
    Render nothing while loading. Native splash is still visible during
    this period, so the user sees the splash, not a blank screen.

PACKAGES USED (already installed by Expo scaffold, no new installs needed):
  - expo-font          ~14.0.11
  - expo-splash-screen ~31.0.13
  - react-native       0.81.5 (provides I18nManager)

ERROR-HANDLING PHILOSOPHY:
  Font load failures don't crash the app — they fall back to system fonts.
  Reasoning: a font failure shouldn't brick a fully-working app. Worst
  case: brand looks generic. Better than worst case: app is unusable.

STARTUP COST:
  Loading 10 fonts adds ~100–300ms to cold start. Could lazy-load lesser
  used variants (Amiri Regular, IBM Plex SemiBold) to shave this, but the
  complexity isn't worth it for ~10 fonts at ~2.3 MB total. Simple > clever.

STEP 4 — RESTRUCTURE lib/theme.ts FOR FONT WEIGHT VARIANTS
================================================================================

FILE MODIFIED: lib/theme.ts

BEFORE (the `fonts` object):
  fonts: {
    arabicDisplay: 'IBMPlexSansArabic',
    arabicBody: 'Tajawal',
    arabicDecorative: 'Amiri',
    englishDisplay: 'CormorantGaramond',
  }

PROBLEM WITH THE BEFORE:
  Each value was a "family name" string with no weight specifier. But
  expo-font knows about 'IBMPlexSansArabic-Bold' as a key, not
  'IBMPlexSansArabic'. Setting fontFamily to the bare family name would
  silently fall back to system font. The before structure was effectively
  unusable.

AFTER:
  fonts: {
    arabicDisplay: {
      regular:  'IBMPlexSansArabic-Regular',
      medium:   'IBMPlexSansArabic-Medium',
      semibold: 'IBMPlexSansArabic-SemiBold',
      bold:     'IBMPlexSansArabic-Bold',
    },
    arabicBody: {
      regular: 'Tajawal-Regular',
      medium:  'Tajawal-Medium',
      bold:    'Tajawal-Bold',
    },
    arabicDecorative: {
      regular: 'Amiri-Regular',
      bold:    'Amiri-Bold',
    },
    englishDisplay: {
      italic: 'CormorantGaramond-Italic',
    },
  }

WHY THIS SHAPE:
  - Each leaf string matches a key in useFonts() exactly. So
    `<Text style={{ fontFamily: theme.fonts.arabicDisplay.bold }}>` works.
  - Nested objects give autocomplete on weights. Type
    `theme.fonts.arabicDisplay.` and editor pops up
    regular | medium | semibold | bold.
  - Per-family weight set matches what was downloaded:
      arabicDisplay → 4 weights (4 IBM Plex files)
      arabicBody → 3 weights (3 Tajawal files)
      arabicDecorative → 2 weights (2 Amiri files)
      englishDisplay → 1 style (italic-only Cormorant)
  - `as const` preserved at the bottom — keeps literal-string typing for
    autocomplete on the values.

NAMING CHOICE:
  Used `semibold` (lowercase, one word) — consistent with Tailwind's
  `font-semibold` and CSS `font-weight: 600` conventions. Alternative
  was `semiBold` (camelCase JS naming).

IMPACT ON EXISTING CODE:
  Zero usages of `theme.fonts.*` in the app prior to this step, so
  restructure was purely additive — nothing broke.

TYPOGRAPHY ASSIGNMENTS:

(1) Brand mark "قهوة"
    Before: fontWeight: '700'
    After:  fontFamily: theme.fonts.arabicDecorative.bold   (Amiri-Bold)
    Removed `fontWeight: '700'` because the weight is baked into the file.

    KEY POINT: Setting both `fontFamily: 'Something-Bold'` AND
    `fontWeight: '700'` in React Native can cause Android to ATTEMPT to
    synthesize an even-bolder version on top of the already-bold font.
    Result: weirdly chunky letters. Best practice: with custom fonts,
    use fontFamily ONLY. Let the file's intrinsic weight speak.

(2) Subtitle "QAHWA"
    Before: fontStyle: 'italic'
    After:  fontFamily: theme.fonts.englishDisplay.italic
            (CormorantGaramond-Italic)
    Removed `fontStyle: 'italic'` for the same reason. The Cormorant file
    IS the italic version. Doubling the slant produces an exaggerated,
    almost-handwritten effect that wasn't desired.

(3) Status indicator (all 3 states: Connecting / connected / failed)
    Added: fontFamily: theme.fonts.arabicBody.regular   (Tajawal-Regular)

CHOICE EXPLAINED:
  Tajawal handles both Arabic and Latin scripts (full Latin coverage), so
  using one font family for all status states keeps the visual cohesive.
  The alternative was using Cormorant Italic for the English status text
  — but switching faces mid-line for small status copy felt scattered.

STEP 7 — RTL LAYOUT (DECISION POINT)
================================================================================

This was the deliberation step. Discussed implications before applying any
code so the choice was made with full context.

WHAT I18nManager.forceRTL(true) DOES:

(a) FLIPS LAYOUT PRIMITIVES
    - flexDirection: 'row' lays out children right-to-left (first child
      visually rightmost)
    - marginStart / paddingStart / marginEnd / paddingEnd swap meaning
    - marginLeft / paddingLeft DO NOT FLIP — they always mean physical
      left. Major footgun: code mixing logical and physical margins
      breaks asymmetrically in RTL.

(b) FLIPS TEXT ALIGNMENT
    - textAlign: 'left' renders right-aligned in RTL
    - textAlign: 'right' renders left-aligned in RTL
    - textAlign: 'auto' uses the text's intrinsic direction (Arabic right,
      Latin left) — usually what you want for mixed content

(c) FLIPS NAVIGATION CONVENTIONS
    - Back arrows in headers point right (← becomes →)
    - Swipe-to-go-back gesture starts from the right edge of the screen

(d) ANIMATIONS DO NOT FLIP
    - transform: [{ translateX: 50 }] always means "physically right by
      50px," regardless of RTL. Animations must be updated manually
      (often: multiply translateX by -1 if RTL is active).

WHY IT REQUIRES A FULL APP CLOSE + REOPEN:
  forceRTL writes to the native side (Android's WritableNativeMap, iOS's
  RCTI18nUtil). These values are READ ONCE AT APP STARTUP when React
  Native sets up its layout engine. JS hot-reload doesn't restart the
  native side, so the flag flip doesn't apply until a real cold start.

  In Expo Go: tapping "Reload" from the dev menu is NOT enough. The user
  must:
    1. Swipe Expo Go away from the recent-apps drawer
    2. Reopen Expo Go from the home screen
    3. Re-scan QR or pick the project from "Recent apps" in Expo Go

  React Native exposes I18nManager.isRTL as a boolean indicating whether
  the cold start has applied the flag yet — useful for "are we actually
  RTL now?" checks.

POLICIES CONSIDERED:
  Option 1: Force RTL always (regardless of device locale)
  Option 2: Force RTL on first boot, persist a flag, never re-toggle
  Option 3: Sync to device locale via expo-localization

  Chose Option 1 for Qahwa specifically: this is an Arabic-first product.
  RTL is a product decision, not a localization preference. Saudi users
  with English-set phones still get Arabic-first layout because that's
  what the product is.

WHAT BREAKS AT THIS STEP (in the home screen specifically):
  Almost nothing — the home screen is centered single-column. The Arabic
  text was already right-to-left intrinsically; the centered "QAHWA" and
  status text don't depend on layout direction. Any visible RTL changes
  will surface later when we add rows of icons, navigation headers, swipe
  gestures, etc.

CODE WRITTEN (added to app/_layout.tsx):
  import { I18nManager } from 'react-native';
  ...
  I18nManager.allowRTL(true);
  if (!I18nManager.isRTL) {
    I18nManager.forceRTL(true);
  }

LINE-BY-LINE EXPLANATION:
  - allowRTL(true): On iOS, RTL is opt-in per app via this flag. Without
    it, forceRTL silently does nothing on iOS. On Android, RTL is allowed
    by default (this is a no-op there). Calling on both platforms is the
    safe cross-platform pattern.
  - The if guard: forceRTL(true) when already RTL works but logs a
    warning. The guard avoids that noise on subsequent boots.

PLACEMENT:
  Right after SplashScreen.preventAutoHideAsync() at the top of the file,
  before the component definition. Both are top-level "configure native
  side at module load" calls; reading them together makes intent clear.

LESSONS LEARNED THIS SESSION
================================================================================

(1) FONT LOADING
    - useFonts() keys must match what fontFamily references. Convention:
      use the filename without extension. Be ruthless about case.
    - Splash-screen gating (preventAutoHide → hideAsync after fonts) is
      the standard pattern for avoiding flash-of-unstyled-text. Worth
      copying for any RN app that uses custom fonts.
    - Loading 10 fonts adds ~100–300ms to cold start. Don't pre-optimize
      with lazy loading until that's actually a measured problem.

(2) FONT FILE STRUCTURE PHILOSOPHY
    - Don't combine fontFamily with fontWeight or fontStyle when using
      custom fonts. The weight/style is baked in. Doing both can produce
      synthesized-bold or double-italic artifacts on Android.
    - Pick fonts for the ROLE they play (display/body/decorative/
      english), not just by what looks pretty. Body fonts need 3 weights
      max. Display fonts need 4. Decorative fonts can have 2 because they
      see limited use.

(3) THEME TOKENS THAT ACTUALLY MAP
    - The previous theme had fonts as bare family-name strings — they
      never could have worked with expo-font, even in principle. Lesson:
      when designing tokens, end-to-end-trace at least one usage path to
      make sure values flow correctly to the platform.
    - Adding semantic colors (success/error) early — even before they're
      used widely — pays off the moment you have 2+ places needing them.

(4) RTL CONFIGURATION
    - It's a NATIVE flag, read once at startup. Hot-reload won't change
      visible RTL state. Always require a full kill-and-reopen during dev.
    - allowRTL(true) is iOS-specific opt-in. Easy to forget and silently
      not work.
    - forceRTL warnings are noisy if you call it when already RTL — wrap
      with `if (!I18nManager.isRTL)`.
    - For Arabic-first products, force RTL unconditionally (don't gate on
      device locale). For products with both LTR and RTL UX intent, use
      expo-localization to detect.

(5) METRO CACHING
    - New assets need `npx expo start --clear`. Without it, you can sit
      with "the file is right there, why isn't it loading" for an hour.
    - --clear adds ~10s to first bundle. Worth it whenever something
      asset-related changes.
